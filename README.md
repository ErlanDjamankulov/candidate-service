# Candidate Service

Микросервис управления кандидатами платформы CV-Scan. Тестовое задание на позицию Java — ведущий программист.

## Стек

- Java 17, Spring Boot 3.5
- PostgreSQL 16, Flyway
- Apache Kafka (spring-kafka)
- Gradle
- JUnit 5, Mockito, AssertJ, Testcontainers, Awaitility
- Springdoc OpenAPI (Swagger UI)
- Lombok

## Запуск

### 1. Поднять инфраструктуру

```bash
docker-compose up -d
```

Поднимаются два контейнера: PostgreSQL на порту 5433 (5432 оставлен на случай локально установленного Postgres) и Kafka на 9092.

### 2. Собрать и запустить сервис

```bash
./gradlew bootRun
```

Swagger UI: http://localhost:8080/swagger-ui.html

### 3. Тесты

```bash
./gradlew test
```

Интеграционные тесты сами поднимают изолированные PostgreSQL и Kafka через Testcontainers — нужен запущенный Docker, но `docker-compose` для них не обязателен.

### 4. Ручная отправка событий в Kafka

```bash
# Одно событие
printf '%s\n' "$(tr -d '\n' < test-events/cv-parsed-sample.json)" \
  | kcat -P -b localhost:9092 -t cv.parsed -l

# 15 событий (14 уникальных + 1 намеренный дубль в конце)
kcat -P -b localhost:9092 -t cv.parsed -l test-events/cv-parsed-bulk.ndjson
```
## Структура проекта

Соответствует структуре из задания: `controller / service / messaging / repository / model / dto / exception`.
## Принятые решения

- **`id` кандидата — строковый, не auto-increment.** В контракте и в Kafka-событии `candidateId` приходит как человекочитаемая строка (`"asanov-bakyt"`), плюс этот же id попадает в URL `/api/v1/candidates/{id}`. Сделал id строкой напрямую. При создании через REST id генерируется на уровне сервиса транслитерацией ФИО в латиницу.

- **Поставил `unique = true` на email и `@Column(nullable = false)` на обязательные поля.** Таблицы создаёт Flyway, но всё равно добавил эти аннотации как дополнительную защиту — чтобы Java-модель и БД были согласованы.

- **`criteria`, `experience`, `questions` — решил сделать просто полем,json - колонкой а не отдельные таблицы,** потому что не увидел смысла делать запросы по содержимому этих данных..

- **`candidateId` в `StatusHistory` сделал текстовым полем, не JPA-связью.** Запрос односторонний: всегда "дай историю кандидата по id". Без связи нет риска N+1. Целостность данных всё равно защищена через FOREIGN KEY на уровне БД.

- **Specification вместо JPQL для фильтрации.** Привычнее было бы сделать через `@Query` с conditional `:param IS NULL OR field = :param`, но в задании Specification упомянут как плюс, поэтому решил углубиться и попробовать новый для себя API.

- **Идемпотентность Kafka-consumer через первичный ключ.** Не стал делать отдельную таблицу дедупликации событий — раз `candidateId` из Kafka это и есть первичный ключ кандидата, повтор события не может создать дубль (сервис проверяет `existsById` и возвращает `false`).

- **`record` для DTO.** Применил новую фичу Java для всех DTO. Удобно — неизменяемость "из коробки", нет сеттеров, короче кода.

- **Producer публикует в Kafka после фиксации БД.** В `changeStatus()` сначала обновление БД и запись в историю, потом `publish` в Kafka. Если бы публикация была первой и потом БД упала, получился бы рассинхрон — внешний мир уже узнал бы про смену статуса, а в БД её нет.

## Что не успел / упрощения

- **DLT (dead letter topic)** не реализован — в задании отмечен как плюс, не обязательно.
- **MDC для логов** не настроил — за рамками обязательной части.
- **`@EntityGraph`/join fetch** — связей `@OneToMany`/`@ManyToOne` между Candidate и StatusHistory нет (история читается отдельным запросом), N+1 в этой схеме не возникает, поэтому EntityGraph не нужен.
- **Тесты** сфокусированы на главных критериях из задания (машина состояний, идемпотентность, фильтрация, коды ошибок). Точечной валидации каждого поля по отдельности нет — есть общая проверка формата ошибки 400.
